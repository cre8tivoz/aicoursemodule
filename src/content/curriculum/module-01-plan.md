# Module 01 — Purpose and posture: static HTML, AI, and the “creative director”

**Course outcomes:** CO-01, CO-04

## Module objectives (AQF 5)

1. Explain in plain language why teams and individuals use **AI plus static HTML** for fast, portable sites.
2. Contrast **creative direction** (goals, audience, constraints) with **mechanical typing** of code.
3. Identify situations where learning fundamentals still changes outcomes (quality, accessibility, maintenance).

## Prerequisites

- None (first module).

## Lessons (5)

### Lesson 01 — What “static HTML” means today

| Phase | Content |
| ----- | ------- |
| **Concept** | A **static** site is files the browser reads as-is (HTML/CSS/images). No server-side database in the learner scope. Good for portfolios, landing pages, documentation micro-sites. |
| **Example** | Compare: a single `index.html` opened locally vs the same file on Netlify/Vercel — same files, different **hosting**. |
| **Practice** | In one paragraph, describe a real work or hobby page that could stay static (no logins). |
| **Summary** | Static = simple to host and fast; you still own quality and accessibility. |

**File:** [`lessons/module-01/lesson-01.md`](../lessons/module-01/lesson-01.md) (Phase 3 content complete).

### Lesson 02 — AI as accelerator, not a substitute for judgement

| Phase | Content |
| ----- | ------- |
| **Concept** | AI can **draft** structure and copy faster than a blank page; it can also be wrong. **Judgement** means checking facts, structure, and ethics before publish. |
| **Example** | Short vignette: AI suggests a hero section; you change heading order for accessibility. |
| **Practice** | List three things you would **always** verify before publishing AI-written HTML. |
| **Summary** | Speed + oversight; you remain accountable for the shipped page. |

**File:** [`lessons/module-01/lesson-02.md`](../lessons/module-01/lesson-02.md)

### Lesson 03 — Creative director: brief, taste, acceptance criteria

| Phase | Content |
| ----- | ------- |
| **Concept** | A **creative director** sets intent: audience, tone, pages, must-haves, “done looks like…”. The IDE and AI are tools under that intent. |
| **Example** | Side-by-side: vague prompt (“make a website”) vs brief (“3 sections, hobby X, mobile-first, WCAG-minded”). |
| **Practice** | Draft a 5-bullet creative brief for a fictional micro-site. |
| **Summary** | Clear briefs reduce rework and improve AI output quality. |

**File:** [`lessons/module-01/lesson-03.md`](../lessons/module-01/lesson-03.md)

### Lesson 04 — Hand-coding, AI-first, and hybrid paths

| Phase | Content |
| ----- | ------- |
| **Concept** | **Trade-offs**: hand-coding builds deep skill slowly; AI-first ships faster but needs literacy to review; **hybrid** is common in practice. |
| **Example** | Table: time-to-first-page, maintainability, learning depth for each path (qualitative). |
| **Practice** | Choose one path for a stated scenario (e.g. personal portfolio in one weekend) and justify in 6–8 sentences (CO-01). |
| **Summary** | No single “right” path—fit depends on goal, risk, and time. |

| Phase | Content |
