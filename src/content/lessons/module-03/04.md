# Review checklist for generated HTML and CSS

AI can produce pages that **look** fine at a glance and still fail **structure**, **keyboard use**, or **meaning** for part of your audience. This lesson gives a **checklist** you can run on every draft. It supports **CO-03**: **evaluate** using foundational HTML/CSS knowledge, not guesswork.

Use it in the **IDE**: you are checking **files**, not screenshots alone.

If you are working from **Gemini CLI** output, **paste** the result into a saved file before you review—do not trust a terminal scrollback as your archive. If you used **Copilot** inline edits, use **diff** view when your editor offers it so you see **exactly** what changed line by line.

## The checklist (print or keep open)

**Page scope**

1. **One clear topic** — Does the main purpose of the page match a single logical `<h1>` (or the first heading that acts as the title)?  
2. **Heading ladder** — Do headings step down without random jumps (for example, `h1` then `h3` with no `h2`)?  
3. **Landmarks** — Are major regions identifiable (`header`, `main`, `footer`, `nav` where appropriate)?

**Links and navigation**

4. **Link text** — Do links describe the destination (“About the club”) instead of “click here”?  
5. **Focus** — Can you tab through interactive elements in a sensible order? (Try **Tab** and **Shift+Tab** in the browser.)

**Content and media**

6. **Images** — Do informative images have `alt` text that carries meaning (empty `alt` only when decorative)?  
7. **Language** — Is `<html lang="en-AU">` (or your language) set appropriately?

**CSS basics**

8. **Contrast** — Is body text readable against the background (quick visual check; you will refine later)?  
9. **Readable size** — Is base font size comfortable on a phone without zooming?

**Process**

10. **Truth** — Are names, dates, prices, and contact details **verified** (AI invents plausible errors)?

### Why these items map to “professional” static sites

- **Headings** are how many assistive technologies **navigate**. Skipping levels is like skipping chapters in a book—listeners lose context.  
- **Landmarks** help users jump to **main** content instead of hearing the header over and over.  
- **Link text** is a small kindness that becomes a big win for people who tab through dozens of links.  
- **Focus visibility** matters for keyboard users; AI-generated CSS sometimes removes outlines—**restore** them unless your design replaces them with an equally clear style.  
- **Contrast** is not only about disability—it is about phones in sunlight and tired eyes after work.

You do not need to be a WCAG lawyer on day one. You **do** need to refuse the idea that “it looks fine to me” is enough.

> **Lighthouse** (used in your capstone) **supports** this checklist; it does not replace **thinking**. Run Lighthouse on the **deployed** URL when you are ready.

### Using the IDE while you check

Keep the HTML file open beside the browser. When you find an issue:

1. **Edit** in the IDE.  
2. **Save**.  
3. **Refresh** the browser.  
4. Repeat.

If you use an assistant, ask for **specific** patches (“change this `h2` to `h1` and move it above the paragraph”) instead of “make it better.” Specificity reduces new errors.

## Example snippet — find the issues

**Draft (do not paste into production):**

```html
<div class="top">
  <div class="logo">My Site</div>
</div>
<div>
  <h2>Welcome</h2>
  <h1>Join our workshop</h1>
  <p><a href="#">Click here</a> for details.</p>
</div>
```

**Problems to spot (there are several):**

- **Heading order** is odd (`h2` before `h1`).  
- **“Click here”** is weak link text.  
- **No `<main>`**, no clear landmark structure.  
- “My Site” might need to be a proper heading or branding decision.

**Your task:** rewrite the structure **on paper or in a file** so `h1` matches the primary topic, link text is descriptive, and you add a `<main>` if appropriate.

## Practice

1. Run the **10-item checklist** on your own **Module 2** `index.html` (or a small AI draft).  
2. **Tick** what passes; **note** one fix you still owe.  
3. If you use an IDE assistant, ask it: “**Review this file against a semantic HTML checklist**,” then **compare** its answer to this lesson—did it miss anything?

### Reflection (AQF-style judgement)

Write **eight to ten sentences** answering: **Which checklist item is easiest to skip when you are tired, and why is that dangerous for real users?** Tie your answer to **at least one** group: keyboard users, screen reader users, or mobile readers. This is not marked for “perfect English”—it is marked for **clear thinking**.

## Key takeaways

- **Evaluate** means **repeatable** checks, not “I like it.”  
- Structure (`h1`, headings, landmarks) is part of **quality**, not decoration.  
- Keep this list next to **Lesson 5** when you loop **generate → check → fix**.

If you want a **single habit** for the next month: every time AI changes HTML, **read the headings first** before you read the colours. That habit will still help when you move to **local models** in Module 4, because structure problems survive every hosting choice.

### Pair review (optional but powerful)

If you can, ask a **peer** to tab through your page without using the mouse. Watch where they get stuck. You do not need to fix everything they mention on day one—you need to **see** what “works for me” missed. Peer review is a low-tech complement to Lighthouse and to your own checklist.

**Screenshot trap:** a PNG of your page **cannot** prove heading order or focus behaviour. Always validate in **markup** and **keyboard** testing, not only in a picture.

If you are colour-blind or reviewing on a poor monitor, **contrast** checks are even easier to get wrong by eye—when in doubt, ask for a second reader or use automated tooling later; do not guess in silence.
