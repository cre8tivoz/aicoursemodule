# Prompt engineering for static sites (without hype)

“Prompt engineering” is a loud phrase for a quiet skill: **say what you need** in a way a model can **use**. For static HTML sites, you do not need mystical phrases or role-play (“You are a 10× developer”). You need **structure** that matches how browsers and humans consume pages: **constraints**, **acceptance criteria**, and **examples** of what “done” looks like.

This lesson supports **CO-04**: **apply** creative direction to tools—cloud or local—so output matches **your** brief, not the internet’s average landing page.

## Introduction

Weak prompts sound like wishes: “Make it nice,” “Make it modern,” “Fix the CSS.” Strong prompts sound like **specs**: file names, technology limits, hierarchy rules, and how you will **verify** the result. The model is not lazy; it is **under-specified**. Your job is to supply the missing **context** from Module 06 Lesson 01’s brief.

Remember limitations from earlier modules: models **hallucinate**, **drift** across long chats, and may ignore organisation policy if you paste secrets. Keep prompts **focused** on one task when possible: “Generate **only** the `<header>` and `<nav>` for `index.html` following…”

## Key concepts

### Constraints

**Constraints** narrow the search space. Examples:

- “**Plain HTML5** and **one** `style.css`—no React, no Bootstrap, no CDN frameworks unless I say so.”  
- “**File names:** `index.html`, `about.html`; **images** in `/images/` with lowercase names.”  
- “**Maximum** three fonts referenced; use **system font stack** unless specified.”

Constraints prevent a common failure mode: you get a lovely snippet that does not fit your folder or your marker’s expectations.

### Acceptance criteria

**Acceptance criteria** are testable statements. They connect directly to review (Module 03 checklist, Module 05 semantics):

- “Exactly **one** `<h1>` on the home page; page title matches brief.”  
- “Every `<img>` has **non-empty** `alt` describing content, or `alt=""` if decorative.”  
- “All internal links work **relative** to project root.”  
- “Navigation appears on **every** page and highlights the current page (class or `aria-current`—your choice, but be consistent).”

When criteria are clear, you can say “not done yet” without arguing taste.

### Examples of “done”

Give **one** concrete shape—even a **wireframe in words**:

- “Hero: full-width background `#1a1a2e`, centred white headline (max two lines), one primary button linking to `#contact`.”  
- “Footer: three columns on desktop, stacked on mobile; include copyright and two footer links.”

You do not need pixel-perfect prose; you need **enough** structure that the model does not invent twelve unrelated sections.

### Before / after — same intent, different clarity

**Weak prompt:**  
“Create a website for my dog walking business.”

**Stronger prompt (illustrative):**

“**Context:** One-page static site for a local dog-walking service; audience is busy professionals on phones.  
**Tech:** Single `index.html` + `styles.css`; no JavaScript unless needed for a simple mobile nav toggle—prefer CSS-only if possible.  
**Sections (in order):** header with business name + nav anchor links; hero with headline + subhead + ‘Book’ button to `#contact`; three-column ‘Why us’; testimonial quote; contact block with fictional phone and email.  
**Constraints:** One `h1`; subsequent sections use `h2`; use `header`, `nav`, `main`, `footer`.  
**Acceptance:** Valid HTML5; all images have alt; contrast readable on white and on the hero background colour I specify: `#2563eb` with white text.  
**Output:** Return **full file contents** for `index.html` and `styles.css` in separate fenced blocks.”

The second prompt is longer because **judgement** was moved from the model’s guesswork into **your** brief.

## Tone in prompts

Match **professional** communication: polite, direct, no insults to the model. Avoid “obviously” and “just”—they add no information. If you need a **revision**, quote what failed: “Replace the hero `h2` with `h1`; keep styling the same.”

## Tools and context windows

In **IDE-integrated** tools (e.g. Cursor-style workflows), you can reference **open files**. Say: “Change **only** `about.html`; do not modify `styles.css` unless required for this page.” In **chat-only** tools, paste the **smallest** relevant excerpt—full files only when necessary—to stay inside context limits and reduce confusion.

### Local vs cloud models (CO-05 reminder)

The **shape** of a good prompt does not change whether the model runs in the browser, on a vendor API, or on your GPU. What changes is **privacy**, **cost**, and sometimes **policy**—topics from Module 04. If you work offline with a local model, your prompts may need to be **shorter** or split into more steps if the context window is smaller. The creative-director habit stays the same: **brief**, **constraints**, **verify**.

### Negative instructions that actually help

“Do not use tables for layout” or “Do not put page titles only in `div` with a class” are **valid** constraints—they reduce common AI habits that hurt accessibility. Pair negatives with a **positive**: “Use a single `h1` for the page title” beats only “don’t mess up headings.”

### When the model ignores part of your prompt

If output violates a stated constraint, **do not** resend a giant prompt. Send a **delta**: “You used Bootstrap CDN; remove it and use only the CSS file I named.” Second attempts work better when you **quote** the offending lines or paste the smallest snippet. This is the same **refine** instinct you will formalise in Lesson 03.

## Practice

**Task — Rewrite the weak prompt**

**Given weak prompt:**  
“Make a cool portfolio page with HTML.”

**Your job:** Rewrite it into a **strong** prompt (150–250 words) that includes:

1. At least **five** explicit **constraints** (tech, files, or structure).  
2. At least **four** **acceptance criteria** tied to semantics or accessibility.  
3. One **short** example of layout or section order.  
4. A line stating **what output format** you want (e.g. fenced code blocks, file names).

**Self-check:** Could a classmate build **without** asking you follow-up questions? If not, add bullets.

## Key takeaways

- **Constraints** + **acceptance criteria** + **examples** beat adjectives like “modern” or “clean.”  
- Tie prompts to your **brief** (Lesson 01) so every generation serves the same contract.  
- **CO-04** is demonstrated when your prompts show **creative direction**, not when they show buzzwords.
