# Where fundamentals still change the outcome

By this point in Module 1, you have a clear idea of what a static site is, how AI can speed drafting, and how a brief sets direction. You might ask a fair question: **If AI can generate HTML quickly, why learn fundamentals at all?**

The short answer: because websites are for **people**, and people do not only experience your page as a picture. They experience it through **structure** (what is a title vs a sidebar), **motion** (keyboard navigation), and **perception** (contrast, readability). Fundamentals are not gatekeeping. They are **quality tools**.

This lesson gives you a grounded “why” and a preview of what comes next. Module **05** in this course goes deeper into reading HTML/CSS and points you to optional practice on [freeCodeCamp](https://www.freecodecamp.org/). Module **02** shifts into **IDE workflow**: folders, files, and editing loops.

## Introduction

Picture two versions of the same visual layout. Both show a header, a main area, and a footer. One version uses a stack of generic containers (`<div>` after `<div>`). Another uses semantic elements like `<header>`, `<main>`, and `<footer>`.

To many sighted people, the two might **look** similar if the CSS is tuned. To many assistive technologies, the difference is not cosmetic. Structure changes how a page is announced, skipped through, and understood.

That is why “I do not need fundamentals” often means “I have not met the user who needed structure yet.” This course is trying to help you meet that idea **before** you ship.

## Key concepts

### Semantics: meaning, not decoration

**Semantic HTML** means: choose elements that describe **meaning**, not only boxes.

Examples (plain language):

- `<header>` — introductory content or navigation grouping (when it matches that role).
- `<main>` — the primary content of the page (typically one per page).
- `<nav>` — navigation links (when you intend that block as navigation).
- `<footer>` — footer information (contact repeats, small print).

Why it matters:

- Screen reader users often jump by landmarks.
- Search engines use structure as a signal (not the only signal, but a real one).
- Future you maintains the site more easily when the file reads like an outline.

### Heading order: the invisible outline

Headings (`<h1>`, `<h2>`, `<h3>`…) create an outline. A common AI mistake is to pick heading levels based on **size styling** instead of **document meaning**. That can “work” visually while confusing assistive tech.

**Risk example:** An `<h4>` appears right after an `<h1>` because the model wanted smaller text. The outline jumps levels. A user navigating by headings loses the thread.

You do not need perfect knowledge on day one. You need the habit: **look at headings as structure**, not font sizes.

### Contrast and readable CSS

Colour choices are not only branding. Low contrast text can exclude people with low vision and can fail basic readability in sunlight on a phone.

CSS fundamentals help you **skim** what matters: text colour, background, font size, line height, spacing. You will practise that in Module 05. For now: treat contrast as part of **quality**, not decoration.

### Maintenance: future you is also a user

A static site might be “done” for launch, then need updates. Hand-off matters:

- If the HTML is semantic, another person can help you faster.
- If the HTML is a soup of anonymous `<div>`s, changes take longer and break more easily.

So fundamentals are not only about accessibility. They are about **sustainable** work.

## Example

### Before/after: same layout idea, different structure

**Before (simplified, div-heavy):**

```html
<div class="top">
  <div class="logo">My Hobby</div>
</div>
<div class="wrap">
  <div class="big">Welcome</div>
  <div class="text">Details here…</div>
</div>
<div class="bottom">Contact</div>
```

**After (simplified, more semantic):**

```html
<header>
  <p class="logo">My Hobby</p>
</header>
<main>
  <h1>Welcome</h1>
  <p>Details here…</p>
</main>
<footer>Contact</footer>
```

**What changed in meaning:** The page now exposes a primary title in `<h1>` and separates header/main/footer roles. A developer (or an AI tool guided by you) can continue without re-guessing intent.

Even this small example connects to accessibility. Screen reader users benefit when the main content lives in `<main>` and the primary title is an `<h1>`. Search tools also prefer pages whose titles read like titles, not mystery boxes.

> **Callout**  
> AI often outputs div-heavy markup first. Your job is to steer it toward **meaning**, not only style.

### A note on free practice (optional, not a hurdle)

If you want structured fundamentals alongside this course, [freeCodeCamp](https://www.freecodecamp.org/) offers free learning paths for responsive layout and basic JavaScript. This course does not require completion of those paths. Think of them as **parallel strength training**—useful, voluntary, and there when you are ready.

Finally, remember that “fundamentals” in this course are not about memorising every HTML element on day one. They are about building **reliable habits**: you glance at structure before you celebrate the design. That habit pairs naturally with AI-assisted speed—without it, speed can become a liability.

## Key takeaways

- Fundamentals connect to **real users** (accessibility), **search clarity**, and **maintenance**.
- Semantic structure is not “extra”—it is part of professional quality for static sites.
- AI drafts often need **structural correction** even when they look pretty.
