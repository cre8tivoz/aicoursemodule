# What static HTML means today

If you work in an office, run a small business, or build a side project, you have probably heard someone say: “We just need a simple page.” In many cases, that simple page does **not** need a complex application. It needs **static HTML**: a small set of files that a web browser can open and show without running custom server code for every click.

This lesson gives you a clear mental model. You will learn what “static” means, where it fits real life, and how hosting works at a high level. You are not expected to memorise every technical term. You **are** expected to explain the idea to a colleague in plain language—which is a useful professional skill at AQF Level 5.

## Introduction

A **website** is often described as if it were one thing. In practice, it is a bundle of **files**. The most important file for our course is usually **HTML** (HyperText Markup Language). HTML tells the browser what the page **is**: titles, paragraphs, links, images, and sections. **CSS** (Cascading Style Sheets) tells the browser how it should **look**: colours, spacing, fonts, and layout. Images and fonts are more files.

A **static** site is a site where those files are prepared **ahead of time**. The server (or your laptop) sends the files as they are. The browser reads them. There is no database in the background changing the page for every visitor—at least not in the simple picture we use in this module.

That sounds basic—and it is. Basic is a strength. Static sites are often:

- **Fast** to load, because the content is ready to send.
- **Cheap or free** to host on modern static hosts.
- **Easy to reason about**, because what you upload is what people get.

They are not perfect for every problem. If you need private accounts, per-user data, or complex workflows, you usually need more than static files alone. For portfolios, event pages, “about us” pages, hobby sites, and many landing pages, static HTML is often enough.

## Key concepts

### Files, not magic

When people say “I built a website,” picture a **folder** on a computer. Inside the folder you might see:

- `index.html` — the home page.
- `about.html` — another page.
- A folder called `css` with `style.css`.
- A folder called `images` with photos.

The browser follows **links** between files. That is the web in its simplest clothes.

### What “static” excludes (in this course’s scope)

This course focuses on **static delivery**. That means we are not teaching you to write server-side programs that build a page on the fly from a database for each request. Many real products use that approach. Your job here is different: learn how to ship **credible, accessible, static pages** with modern tools—including AI—while keeping responsibility for quality.

If your workplace uses a content management system (CMS), the published output might still be **static files** behind the scenes. The idea you are learning still applies: there is a published version people can load quickly.

### Hosting: same files, different location

**Hosting** means: files live on a computer that is reachable on the internet, so other people can request them.

You can open an HTML file **locally** on your machine. Only you see it unless you share the file. When you put the same files on a host such as Netlify or Vercel (topics you will use in the capstone), the public can visit a URL. The **HTML can be the same**; what changes is **who can reach it** and **how**.

> **Callout — Local vs hosted**  
> **Local**: fast for testing; private by default.  
> **Hosted**: public URL; better for real users and for Lighthouse checks on a live URL.

### You still own quality

Static hosting is simple. That does not mean the page is automatically **accessible**, **accurate**, or **on-brand**. The browser will try to show what you uploaded. If the HTML has weak structure or confusing headings, the page can still “work” in a narrow sense while failing real users—including people who use screen readers or keyboard-only navigation.

So: static is a **delivery** choice. **Quality** is a human choice. This course will keep returning to that point.

## Example

Imagine you volunteer for a community group. The group needs:

- A **home** page with the mission and meeting times.
- A **contact** page with an email address and a map link.

You could build:

- `index.html` for the home content.
- `contact.html` for contact content.
- `css/style.css` for shared styling.

You test locally by opening `index.html` in your browser. When you are ready, you upload the folder to a static host. The group shares one public link. Same files—first on your laptop, then on the internet.

Notice what is **not** required for this story: a database, a login system, or a paid “website builder” lock-in. You might still **choose** a builder for convenience. The lesson’s point is: the static model is legible. You can see what you ship.

## Practice

**Task — One paragraph**

In **one paragraph** (roughly 5–8 sentences), describe a **real** work or hobby situation where a **static** site is enough. Include:

- Who the page is for.
- What information it shows.
- Why a login or database is **not** needed.

If you cannot think of an example, use one of these starters and make it specific:

- A personal portfolio for job hunting.
- A weekend event page with date and location.
- A small “services” page for a side business with a phone number and email.

Write in plain language. You are not being marked on fancy words. You **are** practising the skill of stating scope clearly—which matters when you later write prompts for AI tools.

**Self-check (not graded)**

- Does your paragraph name **what changes** if the site becomes non-static? If not, add one sentence.

## Key takeaways

- A **static** site is mainly pre-made **HTML/CSS/assets** served as files—great for many simple public pages.
- **Hosting** decides who can visit; the **files** can stay the same between local test and public deploy.
- **Simple hosting** does not remove your duty to check structure, truth, and accessibility.
- Clear scope (“what this site is for”) helps you later when you direct AI and review output.
