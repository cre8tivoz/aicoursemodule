# Capstone build: ship your static mini-site

Requirements and the marking table live in [`assessments/capstone.md`](../../assessments/capstone.md). You mapped your brief to those rows in [Module 06, Lesson 04](/lessons/module-06/04). This lesson is the **build route** — tick off each stage in order.

Your capstone can be a **new topic** or an upgraded version of **`my-static-site`** — either way, the stages are the same.

The capstone is **not** a hurdle: other modules stay open while you work on it.

## What you are building

**One to three HTML pages**, linked together, **your** topic, hand-authored HTML and CSS, **deployed** to a public URL, with **Lighthouse Accessibility ≥ 90%** on that **deployed** address. No framework required — plain files are fine.

## Stage 1 — Lock the brief

Open your **Module 06** brief. If pages, must-haves, or non-goals are still fuzzy, **edit the brief first**. Scope creep is the main reason capstones slip.

**Check:** Does your brief promise **1–3** pages only? If more, **cut** or **merge** before you build.

## Stage 2 — Files and folders

```text
my-capstone/
  index.html
  about.html          # optional
  styles.css
  images/
```

- Use **relative** links (`href="about.html"`).
- Lowercase file names reduce deploy surprises (`photo.jpg` ≠ `Photo.jpg`).
- Test locally, then again after deploy.

## Stage 3 — Structure before decoration

Build skeleton HTML first: `header`, `nav`, `main`, `footer`; **one** `<h1>` per page; headings without skipped levels. Use real words — not permanent lorem.

Run your **Module 03** checklist: landmarks, heading order, meaningful **alt** text, keyboard path through nav. Fix **structure** before pixel-perfect CSS.

## Stage 4 — CSS that supports reading

Small palette: background, text, one accent. Check **contrast** on text and links. Keep layouts **responsive** — narrow the browser and confirm nothing breaks.

**Do not** remove focus outlines without a visible replacement. Keyboard users need to see where they are.

## Stage 5 — Deploy early

Pick a host you can **justify** in a sentence (free tier, HTTPS, static files, workplace policy). **Netlify**, **Vercel**, and similar hosts are common; your own server works if you have access.

Upload **before** you feel ready. A broken deploy on day one teaches the pipeline; a perfect local folder that never uploads teaches panic on the due date. Re-deploy after each meaningful change.

## Stage 6 — Lighthouse on the live URL

Chrome → DevTools → Lighthouse → **Accessibility** on the **deployed** URL. Save score, date, and Chrome version.

Below 90%? Fix **one** class of issue (contrast, alt text, heading order, focus), redeploy, re-run. **Tab** through the site without a mouse before you call it done.

## Stage 7 — Content and honesty

Write in **your** words. Use images you have **rights** to ship. Be honest about AI assistance — a short note (“Draft from [tool]; structure checked by hand”) is professional.

## Pitfalls (read before you submit)

- **Lighthouse on local files** — scores differ from the live URL; always test deployed.
- **Broken nav after upload** — click every link in a fresh browser session on the live site.
- **Placeholder copy** — lorem ipsum can fail the **Content** row; use real sentences about your topic.
- **Image rights** — copied assets can fail content expectations and create risk outside the course.
- **Feature freeze** — stop adding sections before your accessibility pass; mixing new features with a11y fixes causes regressions.

## Do this — failure rehearsal

In **six sentences**, describe **one** realistic way a capstone could earn **NS** (e.g. broken deploy, Lighthouse 78%, missing `h1`). Then describe the **first** fix you would apply.
