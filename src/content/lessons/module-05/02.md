# HTML semantics in plain English

**Semantics** means **meaning**. In HTML, some elements say “this part is navigation” or “this is the main content.” Other elements—especially generic `<div>` and `<span>`—say only “this is a box” or “this is a span of text,” with **no** role built in.

Assistive technologies (like many screen readers) use semantics to **navigate**. Search engines use semantics as **signals**—not the only signal, but a real one. **You** use semantics to **read** a file and understand it six months later.

This lesson stays practical. You will label markup and rewrite a small block—no exam on memorising every element.

## Introduction

AI tools often output **div-heavy** markup first because `<div>` is a safe, generic container. Your job as creative director is to ask: **What is this section *for*?** If it is navigation, a `<nav>` (or list inside a landmark) communicates that. If it is the primary content, `<main>` helps users jump to it.

You **evaluate** structure here, not only appearance.

## Key concepts

### Landmarks you should know first

| Element | Plain-language role |
| ------- | ------------------- |
| `<header>` | Top-of-page intro: logo, site title, sometimes nav |
| `<nav>` | Block of **navigation links** (not every link on Earth—**major** navigation) |
| `<main>` | The **primary** content of the page—usually **once** per page |
| `<footer>` | Footer: contact repeats, small print, secondary links |
| `<section>` | Thematic grouping when you need a heading inside it |
| `<article>` | Self-contained content (post, card story) when appropriate |

Not every page needs every element. **Wrong** semantics is worse than **boring** semantics—so when unsure, prefer **honest** structure: one `<main>`, sensible headings, and **avoid** pretending a random `<div>` is “main” without `<main>`.

### Div soup versus meaning

**Div soup** means: everything is `<div class="…">` and you cannot tell **what** anything is without reading all the CSS class names.

**Semantic** markup means: even **without** CSS, a human (or assistive tech) can outline the page: header, nav, main, footer.

> Your capstone expects **semantic** HTML and good structure for Lighthouse-style accessibility. Semantics are part of **evidence**, not decoration.

## Example — same layout idea, different meaning

**Div-only (weak semantics):**

```html
<div class="top"><div class="logo">My Site</div></div>
<div class="middle"><div class="title">Welcome</div><p>Text…</p></div>
<div class="bottom">Contact</div>
```

**More semantic (stronger):**

```html
<header><p class="logo">My Site</p></header>
<main>
  <h1>Welcome</h1>
  <p>Text…</p>
</main>
<footer>Contact</footer>
```

**Why it matters:** In the second version, a screen reader user can jump to **main** and hear the **real** page title as a heading. In the first, “Welcome” might be styled like a title but implemented as a `<div>`—a **visual** title without structural truth.

### SEO and semantics (basic, honest picture)

Search engines use **many** signals. Semantic HTML is **one** of them. A clear `<title>`, a single strong topic in `<h1>`, and meaningful link text help **humans** and machines understand the page. No lesson can promise “#1 on Google”—run away from anyone who does. What we **can** say: **clarity** in structure usually aligns with **clarity** for ranking and for accessibility.

### When `<div>` is still OK

`<div>` is not evil. It is **generic**. Use it when no other element fits and you **do not** need a landmark—often for layout wrappers **inside** a semantic region. The anti-pattern is **only** divs everywhere, with meaning carried **only** by class names.

### A note on ARIA (optional teaser)

Sometimes you see `role="…"` or `aria-label="…"` in HTML. Those are **ARIA** attributes for accessibility when native HTML cannot carry enough information. This course stays mostly with **native** elements first (`button` for buttons, real links for navigation). If AI dumps complex ARIA, **pause** and verify against a trusted guide (MDN) before you ship—wrong ARIA can **harm** more than help.

### Lists and buttons (small but mighty)

Use real **`<button>`** for actions that do not navigate away, and **`<a href="…">`** for navigation. AI sometimes styles a `<div>` with a click handler—avoid that pattern for basic static sites; it hurts **keyboard** and **assistive** access. Use **`<ul>` / `<li>`** for real lists of links or items when order matters visually—screen readers announce list length, which helps users.

### Practice reflection (short paragraph)

Write **six to eight sentences** describing **one** time semantic HTML would make a page **easier to maintain** when you return to it in six months—imagine you forgot half the project context.

### Forms and labels (preview for later modules)

Even in static sites you will soon meet **forms**: contact fields, newsletter sign-ups, or simple controls. The semantic pair that matters first is **`<label>`** tied to an **`id`** on the input (`for` on the label matches `id` on the field). Screen reader users hear the label when focus lands in the field; sighted users get a bigger click target when the label wraps or associates correctly. AI sometimes outputs placeholders instead of labels—**placeholders are not substitutes** for accessible names. If you see a bare `<input>` with only `placeholder="Email"`, flag it in review: add a real label or an accessible equivalent you verify against MDN.

### Talking to stakeholders about semantics

Non-technical stakeholders may only care that the page “looks right.” You can translate semantics into **risk** and **cost**: unclear structure means **slower edits**, more **AI mistakes** to fix, and **higher** chance of accessibility complaints or SEO confusion. You are not asking for extra work for its own sake—you are asking for **structure that pays rent** every time someone opens the file or runs an automated check.

## Key takeaways

- **Semantics** = shared meaning between you, browsers, and assistive tech.  
- **Div soup** hides meaning; refactor toward **landmarks** and **real headings**.  
- You evaluate AI output faster when you know **what** to ask for.
