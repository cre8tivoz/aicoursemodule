# Reading code is a professional skill

If you only look at a **preview** of a web page, you see **one** rendering in **one** browser size on **one** day. If you read the **HTML** and **CSS**, you see the **instructions** the browser follows—and you can predict how those instructions might fail for someone else.

This lesson defends a simple habit: **open the file**. That habit is how you graduate from “the AI made something pretty” to “I know what shipped.” That is **judgement** in a technical context: you can explain **why** a page is acceptable, not only that you like it.

## Introduction

Many beginners treat code as a **black box** because AI writes it. Professionals—whether or not they call themselves developers—treat code as **evidence**. Evidence can be wrong. Evidence can be **inconsistent** between files. Evidence can be **inaccessible** while still looking “fine” on your screen.

Reading is also how you **negotiate** with stakeholders. If someone asks, “Can we change the contact block?” you can answer, “Yes—it is this `<section>` in `index.html` and these rules in `style.css`,” instead of, “The AI did something.” That is credibility at work.

Reading code does not mean memorising every HTML element on Earth. It means you can:

- Spot a **missing** closing tag when the layout “looks okay” by accident.  
- Notice a **wrong** heading level when the font size “looks right.”  
- See when **alt** text is missing before you publish.

Those skills make your **prompts** better, too. You stop asking “fix the website” and start asking “close the `<section>` before the footer” or “make the first heading an `h1`.” Specificity is kindness to your future self.

## Key concepts

### Preview lies (sometimes)

Browsers **guess** when HTML is broken. They apply error-correction rules. So a broken tag might still render a layout that **hides** the bug—until a different browser, a screen reader, or a future edit breaks the illusion.

**Example pattern:** an unclosed `<div>` might “work” until you add another section and the DOM nesting shifts. Suddenly the footer jumps or styles apply to the wrong block. Reading the **tree** of tags prevents that surprise.

### AI output is fast, not infallible

Large language models predict likely text. They **do not** “know” your project. They can:

- Close tags in the wrong place.  
- Duplicate `id` values (invalid HTML).  
- Use headings for **styling** instead of **structure**.

**Reading** is how you catch those issues **before** a user does.

### Literacy improves prompts

When you can **name** what is wrong, you can **ask** for a fix in one step. Compare:

- Vague: “Make it more accessible.”  
- Specific: “Use a single `h1` for the page title; keep section titles as `h2`; add `alt` text for the hero image describing the scene.”

The second prompt is the voice of someone who **read** the file.

## Example — broken nesting

**Snippet (simplified, broken):**

```html
<section>
  <h2>Services</h2>
  <div class="grid">
    <article><h3>Design</h3></article>
</section>
```

**What is wrong?** The `<div class="grid">` is never closed; the `</section>` may close too early depending on how the browser repairs the tree. A preview might still look “okay” in one viewport.

**Fix mindset:** match every **open** tag with a **close** in the right order: `article`, then `div`, then `section`.

You do not need to be perfect on the first pass. You need to **look** for the pattern.

## Reading CSS—not line by line, but with purpose

You do not need to memorise every CSS property this month. You **do** need to recognise **patterns**:

- **Selectors** say *which* HTML gets styled.  
- **Curly braces** `{ }` hold **rules** for those selectors.  
- **`property: value;`** pairs set colour, size, spacing.

When AI writes CSS, skim for **surprises**: `position: fixed` on large blocks (can trap content off-screen), `overflow: hidden` on `body` (can break scroll), `!important` everywhere (hard to override later). If something looks wrong in the preview, **search** the CSS file for the class name on that element and read the block **slowly** once.

### Using your IDE

**Split view** helps: HTML on one side, CSS on the other. **Search in file** (`Ctrl+F` / `Cmd+F`) jumps you to `.hero` or `#contact` fast. If your editor shows a **minimap**, use it to spot **repeated** sections—duplicated rules often mean merge opportunities.

### When reading feels slow

That is normal. Speed comes from **repetition**, not genius. Ten minutes of **focused** reading beats an hour of half-watching a video. Keep notes in plain language: “`.btn` makes buttons bold; `max-width` on `.wrap` centres content.” Future you will thank you.

## Practice

**Task A — Find the error**

Below is a deliberate problem. Write **one sentence** naming what is wrong.

```html
<h1>Contact</h1>
<h3>Email</h3>
<p>hello@example.com</p>
```

**Hint:** think **heading levels**, not spelling.

**Task B — Your project**

Open your **practice** `index.html` from Module 2. Skim the first **40 lines**. Write **three** observations:

1. Something you **understand** confidently.  
2. Something you are **unsure** about (a question is enough).  
3. One line you would **ask AI to explain** in plain language.

**Task C — CSS skim (5 minutes)**

Open `css/style.css` (or your stylesheet). **Do not** change anything yet. Write **two** lines: one **rule** you recognise (e.g. “sets font on `body`”) and one **property name** you will look up later (`flex`, `grid`, `margin`—any is fine).

## Key takeaways

- **Preview** is a sample; **source** is the contract with the browser.  
- Reading code supports **debugging**, **accessibility**, and **better prompts**.  
- Small structural mistakes can hide until they do not—**read before ship**.  
- Ten minutes of reading **today** makes tomorrow’s AI edits **cheaper**—because you will see mistakes while they are still small.
